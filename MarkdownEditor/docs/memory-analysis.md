# 整体内存占用分析

图表中进程内存在 **334 MB** 左右波动，属于偏高水平。下面按来源拆解并说明已做与可做的优化。

---

## 1. 主要构成（大致占比）

| 来源 | 说明 | 是否可压 |
|------|------|----------|
| **.NET + Avalonia + Skia** | 运行时、UI 框架、Skia 渲染、字体/主题等 | 难，约 80–120MB+ 是常态 |
| **当前文档与布局** | 文档全文、AST 块列表、布局快照（LayoutBlock/行/Run） | 随文档长度增长，关文档可回收 |
| **图片解码与缓存** | Markdown 内嵌图、预览区/独立窗口的位图 | 已做上限与释放（见下） |
| **公式解析缓存** | 每个不同 LaTeX 公式的 AST（MathNode） | 已加 256 条 FIFO 上限 |
| **多标签/多窗口** | 每个标签一份文档内容 + 可能一份引擎/布局 | 与打开数量成正比 |

---

## 2. 已实施的优化

- **DefaultImageLoader**：图片缓存最多 48 张，FIFO 淘汰并 `Dispose` 对应 `SKBitmap`，避免无限增长。
- **PathToImageConverter**：只保留“当前一张”预览图，路径变化时释放前一张，避免切换预览导致持续上升。
- **ImageViewWindow**：换图前释放旧 `Bitmap`，窗口关闭时释放当前图，避免独立窗口泄漏。
- **MathSkiaRenderer._parseCache**：公式解析缓存上限 256 条，FIFO 淘汰，避免长文档公式过多时无限增长。
- **空文档不创建渲染引擎**：当文档为 0 行或仅 1 行且为空时，不创建 `RenderEngine`/SkiaRenderer/布局引擎/字体，仅绘制背景。**空文档打开时内存可少占约数十 MB**（不再有 200MB 里那部分引擎与字体开销）。

---

## 3. 为何整体仍会到 300MB+？

- **基础盘大**：Avalonia + Skia + 字体（尤其是数学字体如 Latin Modern Math）本身就会占不少内存，这是框架与功能带来的固定成本。
- **文档与布局**：当前打开文档的全文、块列表、布局结果（每块、每行、每个 Run）都会常驻；文档越长、公式/表格/代码块越多，占用越高。
- **GC 与专用字节**：图表里“专用字节”不会在每次 GC 后立刻下降，未用到的堆可能先保留给进程复用，所以曲线会波动且整体偏高，不一定是泄漏。

---

## 4. 若需进一步压内存可考虑

- **缩小图片缓存**：把 `DefaultImageLoader.MaxCachedImages` 从 48 调小（如 24），会多几次重新解码，换一点内存下降。
- **缩小公式缓存**：把 `MathSkiaRenderer.MaxParseCacheEntries` 从 256 调小，公式多的文档会多几次重新解析。
- **用诊断工具精确定位**：用 Visual Studio 的“内存使用率”或 dotMemory 抓一次快照，按“按类型大小”或“按对象数”排序，看是 `SKBitmap`、`byte[]`、字符串、还是布局/文档对象占大头，再针对性优化（例如延迟加载、按需布局、限制历史记录条数等）。

---

## 5. 小结

- 初始 ~200MB、使用后到 334MB，主要来自：**框架与 Skia、当前文档与布局、图片与公式缓存**。
- 图片与公式缓存已加上限并正确释放，避免“越用越高”的无限增长。
- 若仍觉得整体偏大，可先调小上述两个常量，再结合内存快照对最大类型做针对性优化。
